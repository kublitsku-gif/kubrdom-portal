# Портал КубрДом

Портал строительной компании (бани и дома из контейнеров): CRM, объекты, сметы, договоры, снабжение, табель рабочих. Язык проекта и общения — русский.

Прод: **https://portal.kubrdom.ru** (свой домен обязателен: `*.pages.dev` / `*.workers.dev` в РФ режутся по SNI). Админ-панель: `/admin`.

## Архитектура

- **Cloudflare Pages** `kubrdom-portal` — фронт React/Vite (src/ → dist/) + **standalone админ-панель** `public/admin.html` + `public/admin.js` (~700 КБ vanilla JS — основной код панели).
- **Worker** `kubrdom-portal-api` (`src/worker.js`) + **D1** `banya-db` + **R2** `kubrdom-files` (фото/документы) + Telegram-зеркало фото/видео (бот, супергруппа с темами по объектам).
- `functions/api/[[path]].js` (Pages Function) проксирует `/api/*` → Worker на стороне edge — это обход блокировки workers.dev в РФ, НЕ ломать. Клиент ходит same-origin (`API_BASE=""`), файловые URL относительные (`/api/file/<key>`).
- Панель синхронизируется снимком всего стейта в D1 (`storage_key=admin_panel`), автосейв + live-опрос; авторизация заголовком `X-Admin-Token` (секрет `ADMIN_TOKEN` на Worker, fail-closed).

## Деплой

- **Автодеплой: пуш в `main` → GitHub Actions** (Pages + Worker, секрет `CLOUDFLARE_API_TOKEN` комплексный). Ручной wrangler не нужен.
- **Commit-сообщения — ASCII-чистые**: кириллица/грязные байты роняют Pages-деплой (`Invalid commit message [8000111]`).
- Новый секрет в deploy.yml: СНАЧАЛА `gh secret set <NAME>` (хоть плейсхолдером), ПОТОМ пушить код со ссылкой — иначе job падает.
- После `git push` не брать `gh run list --limit 1` сразу (поймаешь предыдущий прогон) — матчить run по `headSha`.
- Секрет `TG_BOT_TOKEN` задан вручную на Worker и переживает деплои — CI его не трогает.

## Сборка админ-панели (не ломать!)

- `npm run build` = `vite build` + `build:admin` (esbuild `public/admin.js` → `dist/admin.js`, target **es2017**).
- `admin.js` — ВНЕШНИЙ файл. НЕ инлайнить обратно в admin.html: >700 КБ инлайн-скрипта мобильный Safari не тянет (белый экран).
- target es2017: `?.`/spread транспилируются, но `async/await` остаётся НАТИВНЫМ. Не опускать target ниже — regenerator-полифил вешал iOS.
- Проверка после деплоя: в задеплоенном admin.js `?.` = 0 вхождений, размер ~874 КБ.
- `<meta viewport>` содержит `maximum-scale=1` (лечит авто-зум iOS на инпутах) — не убирать.

## Критичные правила данных

- **Лимит D1: строка ≤ 2 МБ.** Снимок сохраняется одним транзакционным batch — одна раздутая строка откатывает ВСЁ сохранение. Поэтому файлы (фото, PDF, договоры) ВСЕГДА в R2 ссылками, НИКОГДА не base64 в стейт.
- Загрузка файлов: клиент жмёт фото до ≤1 МБ, Worker принимает img+pdf+office до 20 МБ по allowlist расширений (SVG/HTML запрещены — XSS).
- Частые микро-обновления UI — через `rerenderTab()` (частичная перерисовка `#tab-content`), не полный `render()` (дёргает скролл на iOS).
- Паспортные данные шаблона договора живут ТОЛЬКО в localStorage (`kubr_ctTpl`), в снимок D1 не попадают — так и оставить.

## Гочи процесса

- В репо иногда работают параллельные Claude-сессии: перед коммитом проверять `git diff --stat`, коммитить только свои файлы (не `git commit -am`).
- Стейл-вкладка со старым кодом однажды затёрла облачный стейт (восстанавливали через D1 Time Travel). Против этого работают страж сохранения и «дожим при уходе» в admin.js — при правках зоны сохранения не ослаблять эти механизмы.
